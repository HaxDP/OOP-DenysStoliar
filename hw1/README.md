# god object та принцип єдиної відповідальності (srp)

## вступ

під час розробки програмного забезпечення важливо дотримуватись принципів чистого коду. це робить програму зрозумілішою, простішою в підтримці та розвитку. одним із поширених порушень цих принципів є анти патерн god object, який часто пов’язаний із порушенням принципу srp

## анти-патерн god object

god object - це анти патерн, при якому один клас бере на себе занадто багато відповідальностей. такий клас виконує багато різних задач, знає занадто багато про інші частини системи та керує різною логікою одночасно. у результаті код стає складним для розуміння, важким для тестування та небезпечним для змін

основні ознаки god object:
клас має велику кількість методів  
клас відповідає за різні частини логіки  
будь-яка зміна може вплинути на інші частини системи  
код важко підтримувати та розширювати  

## принцип єдиної відповідальності (srp)

принцип єдиної відповідальності говорить, що клас повинен мати лише одну причину для змін. це означає, що клас має відповідати лише за одну задачу або одну частину функціональності програми. якщо клас виконує декілька різних ролей, то принцип srp порушено

## приклад класу на c++, який порушує srp

```cpp
class userManager 
{
public:
    void createUser() 
    {
        // створення користувача
    }

    void saveUserToFile() 
    {
        // збереження користувача у файл
    }

    void sendEmail() 
    {
        // відправка email користувачу
    }
};
```

у цьому прикладі клас usermanager порушує принцип srp, оскільки він відповідає одразу за декілька різних задач. клас поєднує бізнес-логіку, роботу з файловою системою та відправку повідомлень. будь-які зміни в одній з цих частин вимагатимуть змін у цьому ж класі

## рефакторинг класу з дотриманням srp

для дотримання принципу srp необхідно розділити відповідальності між окремими класами.

```cpp
class userCreator 
{
public:
    void createUser() 
    {
        // створення користувача
    }
};

class userStorage 
{
public:
    void saveUser() 
    {
        // збереження користувача у файл
    }
};

class emailService 
{
public:
    void sendEmail() 
    {
        // відправка email користувачу
    }
};
```

після такого рефакторингу кожен клас відповідає лише за одну задачу. це робить код простішим для розуміння, легшим у тестуванні та зручнішим для підтримки. зміни в одній частині системи не впливають на інші компоненти.

## висновок

анти-патерн god object негативно впливає на якість програмного забезпечення. дотримання принципу єдиної відповідальності дозволяє створювати більш чистий, зрозумілий та підтримуваний код, що є особливо важливим під час розробки великих програмних проєктів.
