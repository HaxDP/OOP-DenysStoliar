# Лекція №24 - Code Smells та рефакторинг: практичний аналіз

## Вступ

У цій роботі я розібрав кілька code smells із власних лабораторних робіт і показав, як вони впливають на підтримку коду. Для прикладу я взяв проєкти `lab1` та `lab2v2`, бо саме там добре видно типові проблеми: змішування відповідальностей, дублювання логіки та надмірне використання рядкових літералів. Навіть коли код працює, він може бути незручним для розвитку, і саме це виявляє рефакторинг.

## Що я знайшов у своїх роботах

У `lab1` видно smell, пов’язаний із жорстко вбудованим HTML і текстом сторінки прямо в обробнику запиту. Заголовок і розмітка знаходяться в тому ж місці, де виконується логіка маршруту, тому представлення та поведінка змішані. Це не критична помилка, але такий підхід швидко ускладнює код, коли сторінка починає рости.

У `lab2v2` проблеми вже помітніші. Обробник `/calculate` робить одразу кілька речей: читає форму, створює матриці, виконує обчислення та ще й збирає HTML-відповідь. Там же є повторюваний код для заповнення `m1` і `m2`, а також магічні рядки на кшталт `m1_00`, `m1_01`, `m2_10`. Саме на цій лабораторній зручно показати практичний рефакторинг.

## Long Method

Перший smell - довгий метод. У `lab2v2` обробник POST-запиту перевантажений різними діями, і через це його важко читати та тестувати. Під час змін у такому методі дуже легко зламати одну частину логіки, навіть якщо інша частина залишається правильною.

Погано виглядає код, де в одному місці одночасно є парсинг форми, обчислення матриць та формування HTML. Краще розділити його на окремі методи: один для читання матриці, другий для обчислень, третій для відображення результату. Після такого розділення кожен блок стає коротшим і зрозумілішим.

Приклад рефакторингу простий: замість одного великого блоку я б залишив у маршруті лише виклик допоміжних методів. Це робить код ближчим до природної мови: спочатку прочитати дані, потім порахувати, потім показати результат.

## Duplicate Code

Другий smell - дублювання коду. У `lab2v2` майже однаково заповнюються дві матриці, змінюються тільки префікси `m1` і `m2`. На маленькому прикладі це ще не здається проблемою, але при додаванні нової розмірності або нової форми довелося б виправляти все в кількох місцях.

Тут добре працює техніка Extract Method. Я б виніс розбір матриці в окремий метод, який приймає форму і префікс. Тоді один і той самий код можна використовувати для обох матриць без копіювання. Після цього помилки в логіці розбору виправляються в одному місці, а не в кількох.

## Magic Strings

Третій smell - магічні рядки. Назви полів форми в `lab2v2` записані прямо в коді. Формально це працює, але такі значення не пояснюють намір. Якщо людина відкриє код пізніше, їй доведеться здогадуватися, що означає `m1_11` і чому саме такий формат.

Кращий підхід - винести формат у зрозумілу константу або хоча б сформувати назву поля через один шаблон. Це не тільки покращує читабельність, а й зменшує шанс помилки при зміні HTML-форми. Тут добре видно, як маленьке перейменування вже допомагає структурувати код.

## Чому тести важливі

Рефакторинг без тестів небезпечний навіть у невеликих проєктах. Наприклад, коли я змінюю розбір матриці на цикл, я можу легко помилитися з межами або переплутати індекси. Така помилка не завжди помітна одразу, але програма почне повертати неправильні значення або аварійно завершуватись.

У випадку з `lab2v2` тест міг би перевірити, що значення з форми потрапляють у правильні комірки матриці. Без такого тесту помилка проявиться тільки під час ручної перевірки або вже в роботі програми. Саме тому рефакторинг треба робити малими кроками і після кожного кроку перевіряти результат.

## Висновок

Мої лабораторні роботи показали три типові code smells: довгі методи, дублювання та магічні рядки. У `lab1` це помітно через змішування HTML і логіки маршруту, а в `lab2v2` - через перевантажений обробник форми та повторюваний код. Усі ці проблеми не ламають програму одразу, але роблять її менш зручною для розвитку.

Рефакторинг допомагає прибрати такі слабкі місця, якщо робити його обережно. Невеликі зміни, зрозумілі назви, виділення методів і тести дають значно кращий результат, ніж велике переписування без перевірки. Саме так код стає не просто робочим, а підтримуваним.
