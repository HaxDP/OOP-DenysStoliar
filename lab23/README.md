# Interface Segregation & Dependency Inversion Principles. Л23
**ISP** та **DIP** *- це принципи SOLID, які допомагають створювати гнучкі та легко тестовані системи з мінімальними залежностями*

---

## різниця між реалізаціями

**поганий ISP та DIP**

*великий інтерфейс з багатьма методами, класи залежать від конкретних реалізацій, склеювання логіки:*

```cpp
class IUserService {
    virtual void saveUser(...) = 0;
    virtual void sendEmail(...) = 0;
    virtual void sendSms(...) = 0;
    virtual void getTotalUsers() = 0;
};
```

**хороший ISP та DIP**

*маленькі спеціалізовані інтерфейси, залежності на абстракції, розділення логіки:*

```cpp
class IUserRepository { virtual void saveUser(...) = 0; };
class IEmailSender { virtual void sendEmail(...) = 0; };
class ISmsSender { virtual void sendSms(...) = 0; };
```

---

## використання

**ISP - Interface Segregation Principle**

- *інтерфейси повинні бути специфічні до потреб клієнта*
- *не змушувати класи реалізовувати невикористовувані методи*
- *краще багато маленьких інтерфейсів, ніж один великий*

**DIP - Dependency Inversion Principle**

- *залежити від абстракцій, а не від конкретних реалізацій*
- *високорівневі модулі не повинні залежати від нижчорівневих*
- *обидва залежать від абстракцій*

*приклад:* `userAccountManager` залежить від інтерфейсів, а не від конкретних класів

---

## переваги і недоліки

**ISP**

$переваги:$
- *гнучкість та модульність коду*
- *легче розширяти функціональність*
- *класи не змушені реалізовувати непотрібні методи*
- *краще розуміння коду*

$недоліки:$
- *більше класів та інтерфейсів*
- *потребує більше планування*

**DIP**

$переваги:$
- *розв'язує залежності між модулями*
- *легше виконати тестування (dependency injection)*
- *спрощує рефакторинг*
- *системи стають масштабованими*

$недоліки:$
- *усім потрібні абстракції (інтерфейси)*
- *потребує більше коду*
- *складніший для початківців*

---

## приклад з лабораторної

*у прикладі показано перехід від поганої архітектури до гарної*

**до:** одна велика служба з усім

```cpp
UserServiceImpl service;
service.saveUser("john");
service.sendEmail(...);
service.getTotalUsers();
service.banUser(...);
```

**після:** інсуляція залежностей

```cpp
userAccountManager manager(
    new mySqlUserRepository(),
    new smtpEmailSender(),
    new twilioSmsSender()
);
manager.registerUser("john");
```

---

## висновок

- *ISP: розділяйте великі інтерфейси на маленькі спеціалізовані*
- *DIP: залежіть від абстракцій, а не від реалізацій*
- *разом вони роблять код гнучким, тестованим та легким у супроводі*
