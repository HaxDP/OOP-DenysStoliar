# Лабораторна робота №22

## LSP - Liskov Substitution Principle.

У цій лабораторній спочатку реалізовано ієрархію, яка порушує LSP, а потім виконано рефакторинг до LSP-сумісного варіанта.

У файлі lab22.cpp базовий клас Printer має метод PrintColorImage, який клієнтський код сприймає як гарантовано робочий для будь-якого об'єкта типу Printer. Похідний клас BlackAndWhitePrinter перевизначає цей метод через виняток, оскільки фізично не підтримує кольоровий друк.

Метод PrintMarketingBrochure демонструє проблему в якій для звичайного Printer усе коректно, а для BlackAndWhitePrinter той самий клієнтський виклик ламається винятком.

## Аналіз порушення

Контракт базового класу такий, PrintColorImage повинен виконувати кольоровий друк. У похідному класі цей контракт фактично скасовано, тому об'єкт підкласу не можна безпечно підставити замість базового типу.

## Рефакторинг

Замість одного широкого типу використано розділення на два інтерфейси: ITextPrinter та IColorPrinter. OfficeColorPrinter реалізує обидва контракти, а OfficeBlackAndWhitePrinter реалізує лише друк тексту.

Клієнтські методи також розділено за відповідальністю: PrintDailyReport та PrintColorPoster. Завдяки цьому кожен клієнт залежить тільки від того контракту, який дійсно потрібен.

## Висновок

LSP гарантує передбачуваність поведінки при підстановці підкласів. У цьому завданні найкращим рішенням стала зміна ієрархії та розділення контрактів на вузькі інтерфейси.
