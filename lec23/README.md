# Лекція 23

## Теорія, практика

Юніт-тестування перевіряє невеликі одиниці коду ізольовано, швидко і автоматично. Його основні переваги це раннє виявлення помилок, безпечний рефакторинг і краща зрозумілість поведінки коду. Водночас юніт-тести не замінюють інтеграційне тестування і не гарантують коректність взаємодії всіх компонентів системи.

## Юніт-тести і інтеграційні тести

Юніт-тести швидкі та ізольовані, тому добре перевіряють локальну бізнес-логіку. Інтеграційні тести повільніші, але вони перевіряють реальні стики з БД, API чи чергами і ловлять проблеми конфігурації. Найкращий підхід це комбінувати обидва рівні тестування.

## Приклад класу з 3 методами і тест-кейсами

Самий самий дефолт що може бути

Приклад класу Calculator:
Add(a, b), Divide(a, b), IsEven(number).

Приклади тестів:

1. Add_SimpleNumbers_ReturnsSum: Add(2, 3) == 5.
2. Add_WithZero_ReturnsSameNumber: Add(0, 7) == 7.
3. Divide_ValidValues_ReturnsQuotient: Divide(10, 2) == 5.
4. Divide_ByZero_ThrowsException: Divide(10, 0) кидає виняток.
5. IsEven_EvenNumber_ReturnsTrue: IsEven(8) == true.
6. IsEven_NegativeOdd_ReturnsFalse: IsEven(-3) == false.

## Для чого і коли треба mock

Mock доцільний, коли клас залежить від зовнішніх систем на кшталт БД, API, файлової системи чи сервісів повідомлень. Він ізолює тест від нестабільних залежностей, дозволяє перевіряти виклики через Verify і задавати контрольовану поведінку через Setup. Якщо метод є чистою функцією без зовнішніх залежностей, зазвичай можна обійтися без mock.

## Висновок

Юніт-тести забезпечують швидкий зворотний зв'язок і якість коду, але максимальний ефект дають у поєднанні з інтеграційними тестами.
Moq варто використовувати там, де є залежності, щоб тести були швидкі, стабільні й передбачувані.
